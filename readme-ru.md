## toolkitrc-report

Language: [EN](readme.md) | **RU**

Утилита для анализа логов зарядного устройства ToolkitRC и генерации
отчётов в формате PDF ([пример отчёта](docs/repoert-example.pdf)).

Разбирает лог-файлы, записываемые зарядными устройствами ToolkitRC
и превращает их в наглядные PDF-отчёты. Отчёты включают параметры зарядки,
статистику по циклам и по всему тесту в целом, графики различных парметров.
заряда и разряда.

Утилита протестирована на файлах, записанных ToolkitRC M8D:
* прошивка **3.01**: старый формат, один файл на весь тест;
* прошивка **3.06**: новый формат, один файл на цикл заряда/разряда.

### Установка

Поддерживается Python версии 3.8+.

```bash
pip install -e .
```

Подключение пакета как точки входа в `setup.py` другого проекта:

```python
entry_points={
    'console_scripts': [
        'toolkitrc-report = toolkitrc_report.cli:cli',
    ],
},
```

### Использование

Утилита поддерживает два взаимоисключающих режима работы:

```bash
# Старая прошивка: все циклы одного теста — в одном лог-файле.
python -m toolkitrc_report -f path/to/log.xls

# Новая прошивка: один файл на цикл, сканируется целая директория
# (включая её поддиректории — см. «Структура директорий» ниже).
python -m toolkitrc_report -d path/to/logs/
```

После установки пакета возможен запуск посредством командной строки:

```bash
toolkitrc-report -d path/to/logs/ -o path/to/reports/
```

Полный список опций:

```
-f, --file FILE     обработать один лог-файл
-d, --dir DIR       обработать директорию с файлами по циклам
-o, --output OUTPUT директория для PDF-отчётов
                    (по умолчанию — рядом со входными данными)
-v, --verbose       выводить в журнал все решения во время анализа
                    (-v) и низкоуровневые детали (-vv)
--log LOG           дополнительно писать лог в указанный файл
--strict            считать любое различие настроек зарядного
                    устройства между файлами (CC/CV/DC/DV и другие),
                    а не только определяющих тест, условием разрыва
                    теста (влияет только на режим -d)
```

`-v`/`-vv` полезны в случаях, когда группировка отчётов,
классификация циклов или отметки «полный цикл» выглядят неожиданно
— каждое неочевидное решение, описанное ниже, логируется на уровне
`INFO` или `DEBUG`. Логгер `matplotlib` всегда отключается, чтобы
вывод`-v` оставался читаемым.

#### Примеры

```bash
# Один лог-файл старого формата, отчёт рядом с ним.
python -m toolkitrc_report -f 2024-09-16_21-23-32.xls

# Директория с файлами нового формата, с логированием решений.
python -m toolkitrc_report -d ./M8D-log -o ./reports -v

# То же самое, плюс сохранение полного лога в файл для дальнейшего
# изучения.
python -m toolkitrc_report -d ./M8D-log -o ./reports -vv --log run.log

# Строгий режим: любое различие CC/CV/DC/DV между файлами,
# а не только различие Mode/DMode/Cyc, разрывает тест.
python -m toolkitrc_report -d ./M8D-log -o ./reports --strict
```

### Входные данные

#### Формат лог-файла

Каждый файл `.xls` — обычный текстовый файл в формате, похожем на
TSV (несмотря на расширение), с четырьмя секциями:

```
==Items==
Type:LiPo Cells:4S Mode:Cycle DMode:Inter CC:500mA CV:4200mV ...

==Data==
time     Vin   Iin  PowerIn  Vout  Iout  PowerCh  Capa  InTmp ...
0:0:1    19750 145  3        16700 -500  8        1     25    ...
...

==End==
Capacity:2265mAh
IntRes1-8S(mou): 0  0  0  0 ...

==Error==
:Exceed safe capacity!
```

- **Items** — параметры зарядки для этого файла: тип батареи,
  количество банок, режим, ток и отсечка по напряжению для заряда и
  разряда, количество циклов и т. д.
- **Data** — одна строка на каждую записанную секунду (интервал времени
  может быть изменён в настройках З/У, используйте минимальный доступный
  интервал для получения наиболее точных данных): напряжения, токи,
  мощность, накопленная ёмкость, температуры и напряжения по банкам (до 8S).
- **End** — добавляется по завершении записи файла; содержит
  итоговую ёмкость и внутреннее сопротивление по банкам.
- **Error** — записывается вместе с `End`; содержит информацию об
  ошибке (например, `Exceed safe capacity!`), либо пометку о
  завершении (`NO ERROR`, `No need`).

Некоторые версии прошивки дублируют секции `End`/`Error`; парсер
обнаруживает и схлопывает дубликат.

#### Варианты формата файлов

|                   | Старый (прошивка 3.01)                | Новый (прошивка 3.03+)             |
|-------------------|---------------------------------------|------------------------------------|
| Циклов на файл    | все циклы теста в одном файле         | каждый цикл в отдельном файле      |
| Именование файлов | произвольное (например, дата и время) | `Ch{N}_{Type}_{Cells}S_{Pass}.xls` |
| Выбирается опцией | `-f`                                  | `-d`                               |

#### Структура директорий (режим `-d`)

Утилита рассматривает как саму указанную директорию, **так и каждую
поддиректорию, содержащую файлы `.xls`**, как независимого кандидата
на сканирование — логи для разных батарей, разложенные по отдельным
папкам, подхватываются автоматически на один уровень вглубь. Внутри
каждого кандидата файлы с одинаковым каналом, типом батареи и
количеством банок группируются и разбиваются на отдельные тестовые
прогоны (см. [Границы тестов](#границы-тестов) ниже).

Именование отчёта зависит от того, что оказалось в директории-
кандидате:

- **один тест** → отчёт называется именем самой директории;
- **несколько тестов с одинаковыми настройками** (одна батарея,
  протестированная несколько раз) → именуются как *последовательность
  тестов*, `<имя_директории>-01`, `<имя_директории>-02`, … по
  порядку номеров прогонов, начиная с 1;
- **несколько тестов с разными настройками** → используется обычное
  именование `Ch{N}_{Type}_{Cells}S_{номера}`, с префиксом в виде
  имени директории для поддиректорий (чтобы одноимённые отчёты из
  разных поддиректорий не перезаписывали друг друга).

Файлы, не соответствующие схеме именования по циклам, обрабатываются
как самостоятельные однофайловые тесты — так же, как в режиме `-f`.

#### Примечание для пользователей зарядного устройства

У зарядного устройства нет встроенных часов реального времени,
поэтому каждый лог-файл создаётся с одной и той же фиксированной
датой — 1980 год: имя файла — единственный надёжный способ понять,
что где находится, так что планируйте тестовые прогоны с учётом этой
особенности. Кроме того, при каждой перезагрузке зарядного устройства
его внутренний счётчик файлов сбрасывается на 1 независимо от того,
сколько файлов с такими же настройками уже есть на SD-карте,
и существующие файлы могут быть незаметно перезаписаны.

> [!WARNING]
> Очистка папки с логами перед началом тестирования — самый
> важный шаг для того, чтобы избежать путаницы и перезаписанных
> или некорректных отчётов.

Порядок действий, позволяющий получить достоверные данные:

- очистите папку с логами на SD-карте зарядного устройства перед
  началом;
- запустите тест или последовательность тестов без перезагрузки
  зарядного устройства между ними (разные каналы можно использовать
  параллельно, поскольку имя файла содержит номер канала);
- скопируйте собранные лог-файлы на компьютер в отдельную папку для
  каждой батареи, используя осмысленное имя директории — оно станет
  именем отчёта (см. «Структура директорий» выше);
- запустите генератор отчётов на этой папке и наслаждайтесь
  подробным отчётом.

### Структура отчёта

Каждый распознанный тест сохраняется в отдельный отчёт в формате PDF,
именованный как описано выше. Структура отчёта:

**Страница 1 — сводка**
- Заголовок, дата/время генерации и список исходных лог-файлов (в
  компактном виде, например `CH0_LiPo_4S_{1-3}.xls`).
- **Параметры зарядного устройства** — таблица с настройками из секции
  `Items` с человекочитаемыми названиями (`CC` → Charge current,
  `PeakV` → ΔPeak voltage и т. д.).
- **Результаты теста** — таблица с общими параметрами батареи: ёмкость,
  энергия, время цикла и количество циклов, отдельно для заряда и разряда.
  Параметры описаны медианным значением и разбросом по циклам; общая
  **эффективность батареи** — медианное отношение энергии разряда к
  энергии заряда по соседним парам полных циклов, показывается только
  если есть хотя бы одна такая пара; плюс общий **статус теста** (см.
  [Статус теста](#статус-теста)) и общее время тестирования.
- **Сводка по тесту** — таблица, содержащая по одной строке на цикл:
  режим, флаг полный/неполный цикл, длительность, начальное/конечное напряжение,
  ёмкость, энергия и статус по файлу, с цветовой подсветкой: зелёный для циклов
  заряда, красный для циклов разряда; последний столбец **Eff, %**
  показывает эффективность каждой пары (см.
  [Эффективность батареи](#эффективность-батареи)), ячейка объединяет
  ровно те две строки, по которым посчитано это значение.

**Страница 2 — графики сводки по тесту** (только если в тесте 3 и
более рабочих цикла, независимо от статуса полный/неполный) —
ёмкость заряда/разряда и энергия заряда/разряда, по одной точке на
цикл «туда-обратно» на общей оси X «номер прохода», начиная с
первого такого цикла. Ёмкость и энергия включают каждый цикл
«туда-обратно» независимо от статуса полный/неполный, поскольку это
измеренные значения, осмысленные в любом случае. Третий график
добавляет эффективность по парам (та же логика объединения в пары,
что и в столбце Eff, %, так что точка есть только там, где оба цикла
пары полные — на остальных проходах на общей оси остаётся пропуск).
Этот график добавляется, только если таких точек не менее двух.

**По одной странице на каждый рабочий цикл** — таблица с
длительностью/напряжением/ёмкостью/энергией плюс графики
напряжения и тока, мощности, ёмкости и энергии; графики температуры
батареи и напряжения по банкам, если такие данные присутствуют в
исходных файлах.

**Две заключительные страницы — обзор всего теста** — те же графики
на нормализованной временной шкале всего теста (все сбросы таймера
зарядного устройства и промежутки между файлами собраны в единую
непрерывную шкалу), с подсветкой периодов заряда/разряда, плюс
входное напряжение/ток/мощность зарядного устройства и его температура
на протяжении всего процесса тестирования.

### Метод анализа данных

Отчёт строится по сырым данным. Для того, чтобы сделать утверждение
«цикл 3 был полным разрядом» или «эта директория содержит данные одной
батареи, протестированной шесть раз», требуется принять ряд
неочевидных решений. Каждое из них логируется в режиме `-v`/`-vv`.
Все метрики вида «значение ± разброс» в отчёте используют медиану,
а не среднее значение, так что один аномальный цикл (например, из-за
ошибочно определённого полного цикла) не искажает итоговое значение
сильнее, чем следовало бы.

#### Границы тестов

Файлы с одинаковым именем канал/батарея/количество банок
упорядочиваются по номеру прогона и разбиваются на тесты. Ожидается,
что тест займёт `2 * Cyc` последовательно пронумерованных файлов в
режиме `Cycle` (один файл заряда и один файл разряда на цикл), либо
один файл для любого другого режима. Новый тест начинается, когда:

- цикл предыдущего файла был прерван возникшей ошибкой (см. ниже),
- предыдущий файл содержит секцию `End` (явное завершение теста),
- достигнуто ожидаемое количество файлов (`2 * Cyc`) — следующий
  файл начинает **новый прогон того же теста** («последовательность
  тестов»),
- нарушена нумерация прогонов (пропуск или сброс после перезагрузки
  зарядного устройства),
- изменились определяющие тест настройки.

Если тест заканчивается меньшим количеством файлов, чем ожидалось
(без ошибки или `End`), его данные помечаются как неполные, а не
считаются полным тестом — это позволяет отлавливать частично
перезаписанные последовательности тестовых прогонов.

«Одинаковые настройки» сравниваются только по параметрам, которые
определяют, *что* представляет собой тест (`Mode`, `DMode`, `Cyc`),
по значению, а не по исходному тексту — `CC`/`CV`/`DC`/`DV`
допустимо подстраивать пользователем между циклами одного теста, и
по умолчанию это не нарушает группировку. С опцией `--strict` любое
различие в любой настройке зарядного устройства, включая
`CC`/`CV`/`DC`/`DV`, разрывает тест, и то же более строгое
сравнение используется при определении, считаются ли настройки
нескольких тестов в директории «одинаковыми» для именования
последовательности тестов (см. «Структура директорий» выше).

#### Классификация сегментов (рабочий цикл или цикл простоя)

Исходный поток данных сначала разбивается везде, где происходит скачок
внутренних часов зарядного устройства — сброс на `0:0:0`,
отрицательный скачок или промежуток, намного превышающий обычный
интервал логирования в 1 секунду (часы зарядного устройства не
вполне выдают стабильные данные, особенно в середине цикла на
некоторых каналах). Каждый получившийся сегмент классифицируется
по следующим правилам:

- сегменты с незначительным изменением ёмкости считаются
  **простоем** (периоды отдыха / таймаута ожидания);
- остальные сегменты считаются рабочими, но только если их
  средний ток составляет не менее 25% от максимального тока сегмента
  в этом же файле — во время отдыха зарядное устройство всё ещё
  логирует небольшой остаточный ток и медленно увеличивающийся счётчик
  ёмкости, так что одного изменения ёмкости недостаточно. Сегмент,
  начинающийся ровно с `0:0:0`, всегда считается рабочим независимо
  от тока, поскольку низкотоковые циклы (долгая фаза дозаряда CV,
  намеренно прерванные циклы) вполне допустимы в некоторых случаях;
- для файлов формата «один файл на цикл» последовательные рабочие
  фрагменты одного направления объединяются обратно в один цикл,
  поскольку сбой часов может разрезать один цикл на несколько
  частей.

#### Определение полного цикла

Цикл считается кандидатом в «полные» заряд/разряд, если он
заканчивается в пределах 2% от заданного напряжения отсечки
(`DV`/`CV` × количество банок) и начинается достаточно близко к
противоположному пределу: в пределах 10% для заряда и 5% для
разряда. Для разряда используется более строгая граница по
начальному напряжению, поскольку это обычно первый цикл теста (без
предшествующего полного заряда в рамках того же теста, от которого
можно было бы оттолкнуться), так что отклонение начального
напряжения на несколько процентов от напряжения отсечки заряда —
более сильный признак действительно неполного цикла, чем такое же
относительное отклонение в начале заряда.

Само по себе данное условие не вполне надёжно — цикл, случайно
затронувший оба предела напряжения, всё ещё может быть неполным.
Например, если он записан сразу после внешнего сброса состояния,
либо если батарея долгое время лежала без использования после
частичного разряда и её напряжение вернулось к номинальному
значению. Поэтому, при наличии не менее трёх проходов в рамках
одного вида, любой из них, отклоняющийся более чем на 25% (по
длительности и/или энергии) от медианы группы, помечается статусом
«неполный».

Статистика медианного значения/разброса на странице сводки использует
только полные циклы, если для данного вида есть хотя бы один такой
цикл, в противном случае — все циклы. Разброс не показывается,
если доступно только одно значение.

#### Эффективность батареи

Строка эффективности объединяет в пары соседние полные
циклы противоположного вида — идущие подряд номера циклов, без
каких-либо циклов (полных или нет) между ними и берёт отношение
энергии разряда к энергии заряда для каждой пары — это эффективность
конкретной пары в круговом цикле, а не значение для отдельного
цикла. Циклы объединяются в пары только в собственном направлении
теста: вид самого первого рабочего цикла (полного или нет) задаёт
это направление для всего теста, так что если тест идёт по схеме
разряд-затем-заряд, полный заряд объединяется в пару только с
предшествующим ему полным разрядом, но никогда со следующим за ним
разрядом. Полный цикл, чей ожидаемый сосед не полон, либо отделённый
от соседа разрывом, остаётся без пары, а не сопоставляется с более
далёким циклом.

Строка показывается только при наличии хотя бы одной такой пары, как
медиана всех значений по парам. Разброс показывается только при
наличии двух и более пар — та же схема, что и для остальных
метрик «медиана ± разброс» в отчёте.

#### Статус теста

Определяется по тому, содержит ли последний файл секцию `End`, была
ли зафиксирована ошибка, прервавшая цикл, и по соотношению ожидаемого
и фактического количества файлов в рамках теста:

- `completed` — нормальное завершение, опционально с
  `; warning: <текст>`, если прошивка записала не критичное
  предупреждение при завершении теста;
- `error: <текст>` — тест был прерван в результате ошибки;
- `incomplete (N of M files)` — найдено меньше файлов, чем ожидалось
  (`2 * Cyc`), без ошибки или `End`, объясняющих причину;
- `incomplete (log truncated)` — у последнего файла вообще нет
  секции `End`.

Текст ошибки засчитывается как прерывание цикла, только если файл
также заканчивается на существенном токе — прошивка записывает
предупреждения (например, «Exceed safe capacity!») и в файлы, чей
цикл всё же нормально завершил фазу дозаряда, так что одного текста
недостаточно как надёжного признака прерывания.

#### Мощность, энергия и временные шкалы

Мощность вычисляется как `Vout × Iout`, а не берётся из
собственного столбца мощности файла, разрешение которого слишком
грубое для осмысленного интегрирования. Энергия вычисляется как
интеграл мощности по времени методом трапеций. Для графиков «всего теста»
относительное время каждого сегмента смещается на общую непрерывную
шкалу, при этом каждый сброс часов схлопывается в один интервал
логирования, так что многофайловые тесты отображаются как единая
непрерывная временная шкала.

### Интеграция

Пакет `toolkitrc_report` можно импортировать напрямую:

```python
from pathlib import Path
from toolkitrc_report import DirectoryScanner, ReportGenerator

for test in DirectoryScanner(Path('./M8D-log')).scan():
    ReportGenerator(test, Path('./reports') / '{}.pdf'.format(
        test.title)).build()
```

Публичный API: 
* `LogFile`, `LogParseError`, `ItemParam` (разбор одного лог-файла);
* `Segment` (один интервал заряда/разряда/простоя);
* `BatteryTest` (одна обнаруженная тестовая программа);
* `DirectoryScanner` (сканер директорий);
* `ReportGenerator` (генератор отчётов);
* вспомогательная функция `format_duration`.
